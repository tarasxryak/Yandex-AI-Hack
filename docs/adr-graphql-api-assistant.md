# ADR proposals: GraphQL API assistant

## Контекст

Команда выбрала продуктовый сценарий: сервис помогает пользователю получить корректный GraphQL-запрос к внешнему API по естественному описанию задачи.

Пользователь пишет, что хочет получить: например, "найди последние заказы клиента с суммой, статусом и товарами". Backend получает или парсит документацию API, Alice AI помогает составить GraphQL-запрос, frontend показывает пользователю запрос, объяснение, возможные риски и предлагает выполнить его.

Дополнительно сервис должен позволять добавлять собственную документацию API в формате PDF или Markdown, чтобы ассистент мог работать не только с заранее известными API.

## Продуктовая формулировка

**Рабочее название:** GraphQL Copilot for API Docs.

**Проблема:** разработчику или аналитику нужно быстро понять чужую GraphQL-схему и собрать правильный запрос без ручного чтения длинной документации.

**Целевой пользователь:** frontend/backend-разработчик, technical support engineer, аналитик интеграций, QA-инженер.

**Ценность:** сервис сокращает время от намерения пользователя до рабочего GraphQL-запроса, объясняет выбранные поля и помогает безопасно выполнить запрос.

## MVP-сценарий

1. Пользователь выбирает API-документацию или добавляет новую.
2. Пользователь пишет задачу естественным языком.
3. Backend получает релевантные части документации.
4. Alice AI составляет GraphQL-запрос и краткую интерпретацию.
5. Frontend показывает:
   - сгенерированный GraphQL-запрос;
   - объяснение, какие поля выбраны и почему;
   - предупреждения, если запрос может быть тяжелым или неполным;
   - кнопку "Выполнить".
6. После подтверждения backend выполняет запрос к API и показывает результат.

## Возможности Alice AI / Yandex AI Studio

- Генерация GraphQL-запроса из естественного языка.
- RAG по документации API.
- Структурирование ответа в JSON: query, variables, explanation, warnings.
- Function calling для поиска документации, проверки схемы и выполнения запроса.
- Работа с файлами: PDF/Markdown документация.
- История запросов и выбранной документации.

## ADR-001: Выбрать продуктовый сценарий GraphQL API assistant

**Статус:** proposed.

**Решение:** делаем сервис, который по пользовательскому запросу и документации API генерирует GraphQL-запрос, объясняет его и предлагает выполнить.

**Почему:** сценарий хорошо попадает в ТЗ хакатона: AI не является просто чатом, а управляет полезным workflow с документацией, RAG, function calling, внешними API и структурированным результатом.

**Альтернативы:**

- универсальный чат по документации;
- генератор REST-запросов;
- low-code конструктор интеграций;
- только GraphQL playground с AI-подсказками.

**Последствия:**

- Нужно сфокусироваться на качестве GraphQL-запросов.
- Нужна обработка документации и поиск релевантного контекста.
- MVP можно показать на одном-двух заранее подготовленных API.

## ADR-002: Использовать React + Go/Gin + SQLite для MVP

**Статус:** proposed.

**Решение:** frontend пишем на React, backend на Go с Gin, данные MVP храним в SQLite.

**Почему:** стек достаточно простой для быстрого прототипа, Go/Gin подходит для API, загрузки файлов и проксирования внешних запросов, SQLite закрывает single-tenant MVP без отдельной инфраструктуры.

**Альтернативы:**

- Next.js fullstack;
- FastAPI + React;
- PostgreSQL вместо SQLite;
- хранить все в памяти без базы.

**Последствия:**

- Быстрый локальный запуск и простая демонстрация.
- Нет сложной инфраструктуры для хакатона.
- Для multi-tenant версии SQLite придется заменить на PostgreSQL.
- Для полноценного векторного поиска позже может понадобиться отдельное хранилище или PostgreSQL с векторным расширением.

## ADR-003: Хранить документацию API как knowledge base

**Статус:** proposed.

**Решение:** загруженную или спаршенную документацию сохраняем в базе как набор документов и чанков. Для каждого чанка храним источник, тип документа, текст, метаданные и связь с API.

**Почему:** Alice AI должна работать с релевантным контекстом, а не со всей документацией целиком. Это уменьшает стоимость запроса, снижает шум и делает ответы объяснимее.

**Альтернативы:**

- каждый раз отправлять весь документ в модель;
- хранить только ссылку на документацию;
- вручную прописывать schema hints в коде.

**Последствия:**

- Нужен pipeline загрузки и разбиения документации.
- Можно показывать пользователю источники, на которые опирался запрос.
- Позже можно заменить простой keyword search на embeddings/RAG.

## ADR-004: Поддержать PDF и Markdown для пользовательской документации

**Статус:** proposed.

**Решение:** MVP поддерживает загрузку Markdown и PDF. Markdown парсим как текст сразу, PDF извлекаем в plain text.

**Почему:** Markdown часто используется в developer docs, PDF часто встречается в корпоративных документациях. Этого достаточно для демо и соответствует ТЗ про работу с файлами.

**Альтернативы:**

- только URL-парсинг документации;
- только Markdown;
- сразу поддерживать HTML, OpenAPI, Postman collections, GraphQL SDL.

**Последствия:**

- Для PDF возможны ошибки извлечения текста.
- Нужны ограничения размера файла.
- Поддержку GraphQL SDL лучше добавить как следующий шаг, потому что это самый точный формат для схемы.

## ADR-005: Разделить генерацию запроса и выполнение запроса

**Статус:** proposed.

**Решение:** AI сначала генерирует GraphQL-запрос и объяснение. Выполнение происходит только после подтверждения пользователя на frontend.

**Почему:** пользователь должен видеть, какой запрос будет отправлен во внешний API. Это снижает риск случайно выполнить тяжелый или нежелательный запрос.

**Альтернативы:**

- сразу выполнять сгенерированный запрос;
- никогда не выполнять запрос, только генерировать;
- выполнять только dry-run валидацию.

**Последствия:**

- UI должен иметь состояние preview -> execute -> result.
- Нужно хранить сгенерированный запрос и статус выполнения.
- Для демо это выглядит как законченный сценарий, а не просто генератор текста.

## ADR-006: Использовать structured output для ответа AI

**Статус:** proposed.

**Решение:** Alice AI должна возвращать не свободный текст, а структуру:

```json
{
  "operationName": "GetCustomerOrders",
  "query": "query GetCustomerOrders($customerId: ID!) { ... }",
  "variables": {
    "customerId": "example"
  },
  "explanation": "Что делает запрос",
  "usedSources": ["docs/orders.md#chunk-12"],
  "warnings": ["Нужен customerId", "Запрос может вернуть много строк"]
}
```

**Почему:** frontend сможет стабильно отрисовать запрос, переменные, объяснение, источники и предупреждения.

**Альтернативы:**

- просить AI вернуть markdown;
- парсить ответ регулярками;
- генерировать только GraphQL без метаданных.

**Последствия:**

- Нужна JSON-схема и валидация ответа модели.
- Ошибки модели можно показывать как retry/repair flow.
- Упрощается сохранение истории.

## ADR-007: Использовать function calling для внутренних инструментов

**Статус:** proposed.

**Решение:** описать для AI набор функций:

- `search_api_docs(api_id, user_request)` - найти релевантные куски документации;
- `get_graphql_schema(api_id)` - получить схему или schema hints, если доступны;
- `validate_graphql_query(api_id, query)` - проверить запрос до выполнения;
- `execute_graphql_query(api_id, query, variables)` - выполнить запрос после подтверждения;
- `save_generated_query(...)` - сохранить результат в историю.

**Почему:** так AI становится управляющей частью сценария, а не просто генератором текста. Это лучше соответствует ТЗ.

**Альтернативы:**

- backend жестко вызывает все шаги по порядку;
- один prompt со всем контекстом;
- ручное управление всеми шагами из UI.

**Последствия:**

- Нужно аккуратно описать контракты функций.
- В MVP можно эмулировать часть функций моками.
- В презентации можно явно показать agentic workflow.

## ADR-008: Для MVP использовать SQLite как single-tenant хранилище

**Статус:** proposed.

**Решение:** SQLite используется для хранения API-документаций, чанков, истории пользовательских запросов, сгенерированных GraphQL-запросов и результатов выполнения.

**Почему:** MVP не требует multi-tenant изоляции, горизонтального масштабирования и сложных миграций. SQLite снижает инфраструктурную стоимость и ускоряет разработку.

**Альтернативы:**

- PostgreSQL;
- in-memory storage;
- файловое хранилище JSON;
- embedded key-value store.

**Последствия:**

- Хорошо для локального демо и одного пользователя.
- Нужно не обещать production-ready multi-user режим.
- При развитии продукта логичный путь миграции - PostgreSQL.

## ADR-009: Парсинг сайтов держать в backend, а не в frontend

**Статус:** proposed.

**Решение:** backend отвечает за загрузку URL, парсинг HTML, извлечение текста и сохранение документации.

**Почему:** frontend не должен работать с CORS, секретами, нестабильными внешними страницами и тяжелым парсингом. Backend может нормализовать документы и переиспользовать их в RAG.

**Альтернативы:**

- парсить сайт в браузере;
- требовать только ручную загрузку файлов;
- использовать внешний crawler-сервис.

**Последствия:**

- Нужны ограничения по доменам/размеру/таймауту.
- Нужно защищаться от SSRF, если пользователь вводит произвольный URL.
- Для MVP можно разрешить только allowlist или заранее подготовленные URL.

## ADR-010: Не исполнять произвольные мутации в MVP

**Статус:** proposed.

**Решение:** MVP генерирует и выполняет только GraphQL `query`. `mutation` либо запрещаем, либо показываем без кнопки выполнения.

**Почему:** mutation может менять данные во внешней системе. Для хакатонного MVP это лишний риск.

**Альтернативы:**

- разрешить все операции;
- разрешить mutation только с дополнительным подтверждением;
- вообще не выполнять GraphQL, только генерировать.

**Последствия:**

- Демо безопаснее и проще.
- Нужно явно валидировать тип операции.
- В будущем можно добавить sandbox/dry-run для mutation.

## Предлагаемая архитектура MVP

```text
React frontend
  - чат/форма пользовательского запроса
  - список API-документаций
  - загрузка PDF/MD
  - preview GraphQL-запроса
  - объяснение, warnings, sources
  - кнопка Execute

Go/Gin backend
  - REST API для frontend
  - upload PDF/MD
  - URL parser для документации
  - chunking документации
  - SQLite repository
  - Alice AI client
  - GraphQL validator/executor

SQLite
  - api_sources
  - documents
  - document_chunks
  - conversations
  - generated_queries
  - executions

External services
  - Alice AI / Yandex AI Studio
  - external GraphQL APIs
```

## Что показать в демо

1. Добавляем API-документацию через Markdown/PDF или заранее подготовленный URL.
2. Пишем: "Собери запрос, который вернет пользователя, его последние заказы и суммы".
3. Сервис показывает GraphQL-запрос, variables, объяснение и источники из документации.
4. Пользователь нажимает "Выполнить".
5. Сервис показывает JSON-ответ и краткую интерпретацию результата.
6. В истории видно прошлые запросы.

## Риски и ограничения

- Если документация плохая или неполная, AI может сгенерировать некорректный запрос.
- PDF может плохо извлекаться, особенно если это скан без OCR.
- Выполнение запросов к внешним API требует настройки auth.
- Парсинг произвольных сайтов несет SSRF-риск, поэтому в MVP лучше ограничить allowlist или показывать это на заранее выбранных API.
- SQLite подходит для single-tenant MVP, но не для серьезной multi-user версии.

## Что можно добавить, если останется время

- Поддержка GraphQL introspection.
- Поддержка GraphQL SDL-файлов.
- Валидация запроса через schema parser до отправки.
- Режим "исправить ошибку GraphQL" по ответу API.
- Сравнение нескольких вариантов запроса.
- Export as curl/fetch/Apollo Client snippet.
- Голосовой сценарий через Алису: пользователь голосом описывает нужные данные.
